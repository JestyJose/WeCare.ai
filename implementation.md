# WeCare Platform Implementation Guide

This document outlines the internal workflow, architecture, tech stack, and component breakdown for the WeCare healthcare monitoring system.

---

## 🔁 System Workflow

WeCare operates entirely natively using an advanced front-end state loop backed by browser `localStorage`. 

1. **Initialization**: The user authenticates via `Login.jsx`, and a session token is written to `localStorage`.
2. **Dashboard Hydration**: `Dashboard.jsx` pulls all historical arrays (Vitals, Wellness, Symptoms, Alerts, Emergency logs, Risk Score) from memory. If data is empty, the system securely seeds 14-months of dummy historical data.
3. **The Smart Risk Engine**:
   - Vitals uploaded via the `AddRecordForm` are evaluated by `VitalMonitor.js` in real-time. If vitals breach warning ceilings, the Risk Score rises by +15. If they breach critical bounds (e.g., BP > 180, SpO2 < 90), it forces the Risk Score to 100.
   - The manual `Check How I Feel` button opens the `WellnessPopup`. Selecting *Moderate* renders a localized ASHA Support Card. Selecting symptoms populates a USP feature: **Intelligent Follow-Up**, which remembers what you logged and queries progression a week later.
4. **Global Alert Subsystem**:
   - Any mathematical shift generated by the Risk Engine is caught by `AlertContext.jsx`.
   - Small shifts trigger non-blocking `ToastContainer` alerts.
   - Warnings spawn `WarningPopup.jsx` in the top right, allowing users to view details.
   - Critical triggers (Severely worsening symptoms, Fall Detection, Risk 100+) instantly spawn the un-dismissible `CriticalPopup.jsx` featuring a 5-second ticking timeline.
5. **Emergency state**: When `CriticalPopup` hits zero, it natively triggers `EmergencyModal.jsx`.
   - The UI is locked, an ambulance siren simulates, and the event is written via `EmergencyManager.js`.
   - The system state shifts into an "Emergency Cooldown", stopping further automated triggers and placing a red banner onto the Risk Monitor screen.
6. **Medical Recovery**: The user clicks "Reset Risk", triggering `ResetRiskModal.jsx`. Confirming medical stability clears the cooldown loop and resets the user's baseline Risk Score back to 10.

## 🛠 Tech Stack

**Frontend Framework**: React.js 18 initialized via Vite.
**Styling**: Tailwind CSS v4.
**Routing**: React Router DOM v6.
**Asset Library**: Lucide React for clean, dynamic SVGs and icons.

**Backend Framework**: Node.js & Express.js.
**AI Integration**: Google Generative AI SDK (`@google/generative-ai`) calling the `gemini-2.5-flash` model.

**Persistence / Database**: Client-side parsing using structured JSON inside standard browser `localStorage` (chosen for immediate, zero-latency 24-hr hackathon deployments without cloud infrastructure).

## 📁 Directory & Component Breakdown

### Backend (`backend/`)

- **`server.js`**: The core AI routing engine. Exposes the `/api/analyze` Express endpoint. Receives symptoms, wraps them in a clinical system prompt to Gemini, enforces strict JSON-return mapping, and delivers the medical assessment back to the client.

### Core Architecture (`frontend/src/`)

- **`App.jsx`**: Wraps the Application Router and houses the `AlertProvider` context boundary.
- **`components/AlertContext.jsx`**: State machine holding `activeAlert`, arrays of queued notifications, `alertHistory`, and active `toasts`. Sorting natively ensures critical popups out-prioritize warnings.
- **`pages/Dashboard.jsx`**: The nervous system of the client application. It manages all top-level states (`riskState`, `records`, `wellnessHistory`, `isCooldownActive`) and conditionally renders all sub-pages.
- **`utils/RiskScoreManager.js`**: Pure functional business logic handling the calculus of Risk Point modifiers, ensuring the Risk Score bounds check.
- **`utils/VitalMonitor.js`**: Numerical parser that identifies when Heart Rate, SpO2, Temperature, or Blood pressure triggers Medical Warning thresholds.
- **`utils/EmergencyManager.js`**: Pure functional logic handling the read/write of automated emergency logs and clearing the boolean `cooldown` flag.

### UI Tracking & Components (`frontend/src/components/`)

- **`SymptomChecker.jsx`**: The frontend consumer payload for the generative-AI feature.
- **`WellnessPopup.jsx`**: Multistep wizard that natively intercepts users logging symptoms. If symptoms existed previously, it shifts into the `SymptomFollowUp.jsx` progression view. When *Moderate* severity is detected, it conditionally surfaces the `AshaWorkerCard.jsx` mock interface safely.
- **`Dashboard UI Overlays`**: Renders `AlertContainer` which hosts `WarningPopup`, `CriticalPopup`, and `ToastContainer` on high z-indexes to guarantee user visibility. Allows `EmergencyModal` sirens inside `Dashboard`.
- **`Log Modules`**: (`WellnessHistory.jsx`, `AlertHistory.jsx`, `EmergencyHistory.jsx`, `SymptomHistory.jsx`) - Secure data tables responsible for parsing locally persisted arrays into readable UI trends for the patient.
- **`AvgVitalsChart.jsx`**: Evaluates the entire payload of raw vitals and mathematically groups them into daily, 7-day, or 30-day averaged numerical cards.
- **`AddRecordForm.jsx`**: Form inputs driving the `handleAddRecord` schema.
